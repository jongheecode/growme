# 그로우미(GrowMe) 설계 문서

## 한 줄 정의
자기계발(공부·독서·운동 등)을 타이머로 몰입 인증하면, 쌓인 시간만큼 캐릭터 "꾸미"가 성장하는 웹 서비스. 미루면 꾸미가 정체되거나 퇴화한다.

## 만드는 사람 / 제약
- 개발자: 컴퓨터공학 전공 학생, 혼자 개발
- 스택: React(웹 프론트) + Node.js(백엔드)
- 기간: 혼자 3~4개월 안에 MVP 출시
- 비용: 무료/저비용 티어 우선, 유료 외부 서비스(특히 유료 AI) 의존 금지
- 오버엔지니어링 금지 — MVP에 맞는 단순 구조, 확장 포인트는 표시만

## 타겟 / 문제 정의
- 타겟: 미루기가 심한 자기계발러(공부·독서·운동). 1차 사용자는 개발자 본인.
- 문제: 기존 습관앱(체크박스)은 실제로 안 해도 눌러버리면 그만이라 검증이 없어 동기부여가 약함. 기존 게임화 앱(펫 키우기 등)은 "진짜 했는지"를 검증하지 않음.

## 핵심 컨셉
1. **실제 인증(엔진)**: 타이머로 몰입한 시간만 기록됨. 체크박스가 아니라 실제 투자 시간이 증거.
2. **꾸미(캐릭터)**: 인증된 시간이 쌓여 꾸미가 성장. 미루면(활동이 없으면) 정체·퇴화.
3. 경쟁 앱과의 차별점: 열품타(공부시간 랭킹 중심, 카테고리 공부 한정) 대비 카테고리 확장 + 시각적 성장·퇴화. SumOne(반려몽 성장)류 대비 "안 하면 퇴화한다"는 리스크 요소로 차별화.

## MVP 범위 (v1)
- 회원가입/로그인: 이메일+비밀번호, Google/Kakao 소셜로그인
- 자기계발 활동(Activity) 만들기: 사용자 정의 이름 + 카테고리 선택(운동/학업/독서/기타 4종 프리셋)
- 타이머 기반 몰입 세션: 시작→진행→종료, 서버가 heartbeat 기반으로 실제 인증시간 계산
- 꾸미 성장: 전체 누적 인증시간으로 성장 단계(크기) 결정, 카테고리별 비중으로 캐릭터 외형(종) 결정
- 퇴화: 2일 연속 무기록 시 3일째부터 게이지 하락 시작, 하루당 일정 %씩 감소, 0 밑으로 내려가지 않음
- 나의 기록 히스토리: 일/주 단위 누적시간, 카테고리별 필터
- PWA로 데스크톱에 "설치형 앱"처럼 사용 가능

## v2로 미룰 것
- 친구와 비교/경쟁, 랭킹, 소셜 기능 (네트워크 의존이라 초기엔 제외)
- 타이머 외 인증 방식 (사진 인증, 외부 연동)
- 화려한 캐릭터 커스터마이징(옷 꾸미기 등), 3D 시각화
- Electron 등 완전한 네이티브 데스크톱 앱
- 알림/푸시, 통계 대시보드 고도화
- 수익화(구독/결제)

## 아키텍처
- **프론트엔드**: React(Vite) SPA. 타이머는 클라이언트에서 카운트, 30초 주기로 서버에 heartbeat 전송. manifest 추가로 PWA 설치 지원.
- **백엔드**: Node.js + Express REST API 단일 서비스.
- **DB**: PostgreSQL 단일 인스턴스(Supabase/Neon/Railway 등 무료 티어).
- **인증**: 이메일/비밀번호(자체 세션 or JWT) + Google/Kakao OAuth.
- **배포**: 프론트 Vercel/Netlify, 백엔드 Railway/Render 무료·저비용 티어.
- **캐릭터 에셋**: 카테고리×성장단계 조합 수만큼의 정적 이미지, 프론트에서 정적 파일로 서빙.

## 데이터 모델

**User** — id, email, password_hash(nullable), oauth_provider(google/kakao/null), oauth_id, nickname, created_at

**Activity** — id, user_id, name, category(enum: EXERCISE/STUDY/READING/ETC), created_at

**Session** — id, activity_id, user_id, started_at, ended_at, verified_seconds, created_at

**Growth** (유저당 1행) — id, user_id(unique), current_gauge, last_active_date, updated_at

설계 포인트: "누적 인증시간"(Session 합산, 히스토리용, 절대 감소하지 않음)과 "현재 성장 게이지"(Growth.current_gauge, 퇴화로 감소 가능)는 별도 값으로 관리한다. 지배적 카테고리(캐릭터 외형 결정)는 Session을 Activity.category 기준으로 그룹핑해 계산하며, MVP 단계에서는 별도 집계 테이블 없이 쿼리로 처리한다.

## 꾸미 성장 로직
- **외형(종)**: 유저의 카테고리별 누적 인증시간 중 가장 큰 비중을 차지하는 카테고리로 결정. 비중이 바뀌면 외형도 바뀔 수 있음.
- **크기(성장 단계)**: 전체 카테고리 합산 누적 인증시간(Growth.current_gauge)으로 결정 — 특정 카테고리만 본 게 아니라 어떤 자기계발이든 인정. 카테고리(4종)마다 알 포함 5단계 이미지 세트를 갖고, current_gauge 구간에 따라 단계가 정해짐(정확한 시간 구간 값은 구현 단계에서 밸런싱).
- **퇴화**: 2일 연속 무기록 시 3일째부터 current_gauge 하락 시작, 하루당 일정 %씩 감소, 0 밑으로는 내려가지 않음(리셋 없음). 매일 자정 배치(`node-cron`)로 계산.

## 유저 플로우
1. 랜딩/로그인(이메일 or Google/Kakao)
2. (최초 1회) 온보딩 — 첫 활동 만들기
3. 홈(대시보드) — 꾸미 캐릭터 + 게이지 + 오늘 누적시간 + "타이머 시작"
4. 활동 선택/생성
5. 타이머 진행 — 탭 이탈 시 자동 일시정지, 복귀 시 재개
6. 세션 종료 — 인증시간 결과 + 게이지 반영
7. 히스토리 — 일/주 단위 누적시간, 카테고리별 필터
8. 설정 — 프로필, 계정 연동, 로그아웃

플로우 요약: 홈 → 활동 선택 → 타이머 → 결과 → 홈 복귀.

## 타이머 검증 방식
탭이 백그라운드로 가거나(다른 탭 전환, 창 최소화) 일정 시간 입력이 없으면 타이머 자동 일시정지, 복귀 시 재개. 클라이언트가 30초마다 서버에 heartbeat를 보내고 서버가 누적시간을 계산 — 최소한 "그 화면을 보고 있었다"는 검증.

## 에러 핸들링 / 엣지케이스
- 네트워크 끊김: 클라이언트가 로컬로 타이머 유지, 재연결 시 sync. heartbeat 5분 이상 끊기면 서버가 마지막 heartbeat 시점 기준으로 세션 자동 종료.
- 브라우저 강제 종료/크래시: 마지막 heartbeat까지만 인증시간으로 인정.
- 활동 삭제: 과거 세션은 유지, 히스토리에 "삭제된 활동"으로 표시.
- 소셜/이메일 계정 중복: 자동 병합 안 함(별개 계정 취급), 계정 연결은 v2 확장 포인트.
- 퇴화 계산: 매일 자정 배치, 별도 인프라 없이 백엔드 프로세스 내 스케줄러로 처리.

## 테스트 방침
핵심 로직(게이지 증가/퇴화 계산, 세션 유효시간 계산)만 단위테스트로 커버. UI는 로그인 → 타이머 → 결과 확인까지의 최소 흐름만 수동/e2e로 확인.
