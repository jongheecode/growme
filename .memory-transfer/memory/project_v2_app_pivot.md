On 2026-07-17 the user decided to pivot GrowMe from a timer-first web PWA to a **React Native app** built around a Tamagotchi-style companion loop. This supersedes the previous plan queue — see [[project_design_asset_integration]] (abandoned) and [[project_v15_desktop_shell_profile]] (backend kept, web frontend abandoned).

**Core loop (confirmed via brainstorming, not yet spec'd):**
- App opens into a free-form AI chat with 꾸미 (Kkumi) — no fixed script, the conversation naturally leads the user to set a goal.
- Goal-setting produces tasks ("미션") two ways: AI suggests tasks from the goal, AND the user can add their own directly (hybrid, not one or the other).
- Each task carries an AI-assigned XP reward (AI looks at difficulty/estimated time and sets XP, e.g. "리스닝 20분" → +15XP) — completing a task shows "+10XP"-style feedback and accumulates toward Kkumi's level/growth stage. This replaces the old time-accumulation `Growth.currentGauge` model entirely — growth is now XP-from-task-completion, not verified timer seconds.
- Tasks auto-fail if their deadline (today/this week) passes uncompleted — no manual "give up" button.
- Kkumi reacts to every completion/failure with an AI-generated (not scripted) contextual line each time.
- Timer is NOT a secondary/detached feature — it lives inside a task's execution screen: for missions that need timing, the user starts a timer scoped to that specific task while the app is open, to measure the work as they do it. Think "mission detail screen has an optional timer," not "global timer tab."
- Existing category concept (운동/공부/독서/기타) carries over via each task's category, still expected to influence Kkumi's visual mix/appearance the way it does today.

**Home screen (validated with the visual companion, mockup files in `.superpowers/brainstorm/825-1784226112/content/`):** Kkumi sits large, centered, in a house/room-style background (Option "A. 돌봄 홈" — the most Tamagotchi-like of 3 options shown). Tasks are NOT a bottom tab — a small floating circular button shows progress (e.g. "2/4") and, on tap, morphs (grows in place, ~0.38s, container-transform style) into a bottom sheet listing today's tasks; tapping again or the dimmed backdrop reverses the animation. Bottom tab bar is just 홈/히스토리/프로필. Colors/exact visual polish in that mockup are explicitly placeholder — see [[feedback_design_workflow]], the user builds final visuals elsewhere.

**Kkumi variety + personality (added 2026-07-17, implemented in sub-project 2 — see below):**
- Species: one of 3 possible Kkumi species is assigned **randomly, once, at the moment of first XP** (task completion) — an "egg" state before that. Species never changes afterward. This was refined mid-brainstorm from an earlier "cumulative pattern decides species" idea — the user explicitly wanted the hatch to be random but ongoing appearance/personality to drift toward resembling the user's own behavior over time (motivational "내 꾸미가 나를 닮아간다" framing). As implemented in sub-project 2: species is the one-time random draw; the **personality** layer is what does the ongoing drift, computed live from behavior (not stored), so it naturally updates as the user's patterns change.
- Personality is MBTI-style: **2 axes → 4 types**, computed from mission completion history only for now (컴퓨티드 from `Task` rows) — 꾸준함↔느슨함 (completion rate) and 여유↔막판형 (how early vs. last-minute tasks get completed). Conversation-data-based refinement is explicitly deferred to sub-project 3+ once AI chat exists; the calculation function is designed to have more signals added later without restructuring.
- Net effect: two users with the same XP total can have different species (RNG) and different personality types (behavior-derived) — a deliberate design goal for curiosity/shareability, not an edge case to eliminate.

**Platform decision:** React Native full rewrite (not Capacitor-wrap) — user explicitly chose this knowing it means redoing the Tailwind+SVG web UI (including the `renderKkumi` SVG character work) as native components. Backend (Express+Prisma: auth, profile CRUD, `bio`, cascade deletes) is reused as-is; only the `frontend/` web app and the `html/` mockup-port plan are abandoned.
